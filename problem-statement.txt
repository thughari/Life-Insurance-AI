1. Project Overview
Group 03 is tasked with building a production-grade Life Insurance AI Copilot — an intelligent assistant that handles
the full lifecycle of a life insurance interaction: from initial applicant risk intake and underwriting Q&A;, through policy
type selection guidance, beneficiary nomination support, and policy issuance status tracking.
The key technical differentiator for this project is the use of a LangGraph stateful workflow — a directed graph
where an intent router node conditionally branches incoming queries to one of four specialist agent nodes
(Underwriting, Policy Q&A, Beneficiary, Issuance), each with access to the shared application state and the RAG
knowledge base. High-risk underwriting applications additionally trigger a Human-in-the-Loop review node before
proceeding.

2. Problem Statement
Life insurance is one of the most complex insurance products to explain, sell, and administer. Applicants are often
confused about which policy type suits their needs (term vs whole vs ULIP), what health disclosures are required
during underwriting, how risk scoring affects their premium, and how to correctly nominate a beneficiary. Advisors and
underwriters spend hours handling queries that could be resolved instantly with the right information surfaced at the
right time.
Unlike other insurance domains, life insurance queries are not isolated — they follow a sequential, stateful journey. An
applicant who starts by asking about term vs whole life will next ask about premium calculation, then health disclosure
requirements, then beneficiary nomination, then issuance timelines. A stateless chatbot fails this user at every step. A
stateful, graph-based agent that remembers context across nodes is the right architecture for this domain.
■ Your Mission
Build a LangGraph-powered AI copilot that routes each user query to the correct specialist node based on intent,
maintains shared application state across all interactions in a session, retrieves accurate information from policy
documents, and — for high-risk applicants — pauses the workflow and routes to a human underwriter for review
before issuing a recommendation.
The system must demonstrate conditional graph branching, state persistence across nodes, RAG retrieval from life
insurance documents, and an observable multi-node trace in LangSmith.
2.1 Key Pain Points Your System Must Address
• Policy Type Confusion: Applicants cannot easily distinguish between Term Life, Whole Life, Endowment, and
ULIP products and which suits their financial goals.

• Underwriting Opacity: Applicants don't know what health conditions, lifestyle factors, or financial disclosures will
affect their risk score and premium.
• Premium Estimation Gap: Advisors manually look up actuarial tables to estimate premiums — a slow, error-prone
process.
• Beneficiary Nomination Errors: Policyholders make errors in beneficiary nomination (incorrect share allocation,
missing nominees) that cause claim disputes.
• Policy Issuance Delays: Applicants don't know what documents are pending, what the issuance timeline is, or
why their application is under review.
• High-Risk Application Handling: Borderline or high-risk applications require human underwriter review — the
system must know when to escalate rather than auto-respond.
• Lapse & Revival Queries: Policyholders who have missed premiums need guidance on revival conditions, grace
periods, and reinstatement requirements.
■■ Non-Negotiable Safety Constraint
The copilot must NEVER provide a final underwriting decision, medical advice, or guaranteed premium quote. All
premium estimates must be labelled as indicative. High-risk cases MUST be routed to the Human-in-the-Loop
review node — auto-approving them without human review is a critical system failure. Every policy-grounded
answer must cite the source document and page.

The two CSV datasets (premium rates and risk scores) are structured lookups — think carefully about whether to
index them in FAISS or handle them as direct lookup tools in your graph nodes. The shared LangGraph state
should carry applicant data collected across nodes (age, health disclosures, desired cover, policy type preference)
so each specialist node can access what prior nodes captured.

4.2 Required Data Flow (5 Steps)
• Step 1 — Query Input: User enters a query in the Streamlit UI. Session state carries prior context (applicant data
collected in previous turns).
• Step 2 — HTTP POST /chat: FastAPI receives query + session_id, validates, and passes to the LangGraph
workflow executor.
• Step 3 — Graph Execute: The Intent Router node classifies the query and conditionally edges to the correct
specialist node. The node executes with access to shared state.
• Step 4 — RAG Retrieve / Structured Lookup: The active node retrieves relevant chunks from FAISS (for
document queries) or looks up the CSV tables (for premium/risk score queries). State is updated with new
applicant data collected.
• Step 5 — Respond & Persist: The LLM synthesizes a cited response, guardrails check the output, and the
updated state is persisted via checkpointer. Response returned to user.

5.1 Non-Functional Requirements
• Performance: API must respond within 4 seconds at the 95th percentile.
• Graph Observability: Every graph execution must produce a LangSmith trace showing node-by-node execution,
conditional edge taken, and state delta at each node.
• State Durability: Shared state must persist via LangGraph checkpointer — session data must survive API restarts
within a demo window.
• Security: All credentials stored as environment variables. No hardcoded secrets.
• Portability: Full application must run via a single Docker command.
